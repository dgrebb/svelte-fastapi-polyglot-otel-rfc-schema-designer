# Agent guide — svelte-fastapi-remote-functions

This repo is a **polyglot monorepo** (`api/` + `ui/`) designed for parallel human and LLM development. Treat each side as an independent product that happens to share a git root.

## Layout

| Path | Role | Version file | Release tag prefix |
|------|------|--------------|-------------------|
| `api/` | FastAPI + SQLAlchemy | `api/pyproject.toml` → `project.version` | `api-X.Y.Z` |
| `ui/` | SvelteKit + remote functions | `ui/package.json` → `version` | `ui-X.Y.Z` |
| Root | Compose, Makefile, git hooks, agent docs | — | — |

**There is no `v` prefix on tags.** Tags look like `api-0.2.0` and `ui-0.3.1`, never `v0.2.0`.

## Worktrees & parallel work

Developers (and agents) are expected to use **git worktrees** for concurrent streams:

- `feat/api-*` worktrees → touch `api/` primarily; scope commits `api`
- `feat/ui-*` worktrees → touch `ui/` primarily; scope commits `ui`
- Cross-cutting changes (compose, docs, hooks) → scope `compose`, `docker`, `repo`, or `ci`

Keep changes **scoped to the task**. Avoid drive-by edits on the other package unless the task requires it.

When both packages must change (e.g. new API field + UI form), prefer **two linked PRs** from separate worktrees, or one PR with clearly separated commits per scope.

## Commits & pull requests

We use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) with one extra type:

```
<type>[optional scope]: <description>
```

**One line only** — subject line is the whole commit message. No body or footer unless you have a rare, explicit reason (e.g. `BREAKING CHANGE:` that cannot fit in the subject). Put implementation notes, screenshots, and rationale in the **PR description**, not in git history.

On feature branches and worktrees, short one-line commits are fine even if imperfect; they are squash-merged away. Use a longer body only when it helps **you** during review on that branch — not for `main`.

### Types

| Type | Use |
|------|-----|
| `feat` | New capability (MINOR bump when released) |
| `fix` | Bug fix (PATCH bump) |
| `docs` | Documentation only |
| `style` | Formatting, no logic change |
| `refactor` | Structure change, no behavior change |
| `perf` | Performance improvement |
| `test` | Tests only |
| `build` | Build system / packaging |
| `ci` | CI configuration |
| `chore` | Maintenance, tooling |
| `revert` | Reverts a prior commit |
| **`wip`** | **In-progress work** — not release-ready; used to mark beta / unreleased / pre-release streams in history |

**Only `type` and `description` are required.** Scope is optional but strongly encouraged.

### Scopes

| Scope | When |
|-------|------|
| `api` | FastAPI app, models, routers, API tests |
| `ui` | SvelteKit app, components, remote functions |
| `compose` | `compose.yaml`, `compose.dev.yaml` |
| `docker` | Dockerfiles, `.dockerignore` |
| `docs` | README, AGENTS, changelogs |
| `deps` | Dependency bumps (pip, pnpm) |
| `ci` | GitHub Actions, hooks automation |
| `repo` | Root Makefile, `.cz.toml`, `.githooks`, `.cursor/agents`, shared tooling |

Examples:

```
feat(ui): stream status metrics with query.live
fix(api): validate workflow step order on create
wip(api): experiment with task retry policy
chore(deps): bump fastapi to 0.115.2
docs(repo): clarify worktree workflow in AGENTS
```

### Squash merges

**All PRs are squash-merged.** The **PR title** (one line) becomes the permanent commit on `main`. It must pass Commitizen validation (`cz check`).

- **Title** — single conventional line, e.g. `feat(ui): add live status dashboard`
- **PR body** — details, screenshots, test plan, breaking-change notes (`BREAKING CHANGE:` here if needed)

Local commits inside a PR can be one-line and messy; only the squash title survives on `main`.

### Git hook

`cz check` runs on `git commit` via `.githooks/commit-msg` (Commitizen, Python). Install once:

```bash
make install-hooks
```

Manual check of the last commit: `make check-commit`

## Releases (independent per package)

`api` and `ui` **version and release independently**. Commitizen ([Python](https://commitizen-tools.github.io/commitizen/)) bumps semver from conventional history.

```bash
make bump-api   # tags api-X.Y.Z, updates api/CHANGELOG.md
make bump-ui    # tags ui-X.Y.Z, updates ui/CHANGELOG.md
```

- `feat` / `fix` / `BREAKING CHANGE` drive semver bumps
- `wip` commits do **not** trigger a release bump (they signal in-progress / pre-release work)
- Release commits are generated by Commitizen (`chore(api): release api …`)

Future automation can map `wip`-heavy lines to `beta`, `pre-release`, or `unreleased` channels without changing tag format.

## Subagents

Specialist agents live in **[`.cursor/agents/`](.cursor/agents/README.md)**. Delegate atomic work to preserve main-thread context.

| Subagent | Scope |
|----------|--------|
| `openspec-pm` | OpenSpec changes, task breakdown, multi-agent coordination |
| `svelte-file-editor` | `ui/` — Svelte 5, SvelteKit, `query.live`, shadcn ([Svelte AI subagent](https://svelte.dev/docs/ai/subagent)) |
| `fastapi-backend` | `api/` — FastAPI, SQLAlchemy, pytest |
| `unit-test` | pytest (`api/`) + Vitest (`ui/`) |
| `e2e-test` | Playwright browser tests (`ui/` only) |
| `docker-compose` | Dockerfiles, compose, Makefile |

**`openspec-pm`** may assign tasks and spawn other subagents without waiting on the orchestrator, but must **keep the main thread informed** and relay user input between agents.

Svelte MCP instructions: [`.cursor/instructions/svelte.md`](.cursor/instructions/svelte.md) ([Svelte AI overview](https://svelte.dev/docs/ai/overview)).

## What agents should do

1. **Read the scope** — stay in `api/` or `ui/` unless the task is cross-cutting.
2. **One-line conventional commits** — `type(scope): description` only; save detail for PR descriptions.
3. **Use `wip`** for exploratory or incomplete work that should not imply a release.
4. **Do not add `v` to version tags.**
5. **Do not couple releases** — bumping the API does not require bumping the UI.
6. **Prefer existing patterns** — remote functions in `*.remote.ts`, FastAPI routers in `api/app/routers/`, shadcn components in `ui/src/lib/components/ui/`.
7. **Keep the example exemplary** — clear names, minimal magic, no stray scaffold or agent notes in shipped code.

## What agents should avoid

- Monolithic commits touching `api/` and `ui/` without reason
- Custom SSE endpoints when `query.live` suffices
- API endpoints solely to feed UI status (UI status is self-contained)
- Committing secrets, `.env`, `data/`, `node_modules/`, `.venv/`
- Renaming “Agent Orchestrator” in the UI unless asked (product name ≠ repo name)

## Quick commands

```bash
make dev              # docker compose dev (api + ui)
make up               # docker compose production
make install-hooks    # api venv + Commitizen git hooks
make check-commit     # validate last commit message
make bump-api         # release api
make bump-ui          # release ui
cd api && make test   # pytest
cd ui && pnpm check   # svelte-check
```
